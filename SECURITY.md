# Security policy

OpsMind touches three surfaces that a security-conscious reader cares
about: generated SQL that runs against a live database, an LLM that can
be jailbroken, and an audit log that regulators rely on. This document
explains what we guarantee and how to report a problem.

## Supported versions

OpsMind is deployed, not released. The supported version is **`main`**.
Every push to `main` passes the CI lint + test + eval-library jobs before
it lands. If you are running an older commit, upgrading to `main` is
always the first recommended fix.

## Threat model

OpsMind's security story is in `specs/security.md`. In short:

- **SQL safety** — `modules/sql_validator.py` enforces a five-stage
  pipeline (statement type, injection patterns, table existence, column
  existence, row-limit injection) on every LLM-generated query. Coverage
  lives in `tests/unit/test_sql_validator.py`.
- **Read-only posture** — only `SELECT` and `WITH` statements execute.
  `INSERT`/`UPDATE`/`DELETE`/`DROP`/`ALTER`/`TRUNCATE`/`EXEC` keywords
  fail validation before reaching the database.
- **Local models** — Ollama runs in-process or in a sibling container.
  No request leaves the factory network by default.
- **Audit log** — every agent step emits a JSONL event to
  `logs/audit.jsonl` with a UTC timestamp. Covers the BRC 8.5.1
  electronic-records requirement.

Known gaps are documented:

- `tests/unit/test_sql_validator.py::TestInjectionPatterns
  ::test_embedded_line_comment_with_benign_content_blocked` — xfail
  marking a defence-in-depth gap in the comment-injection regex. Not
  exploitable via stage-1 keyword checks, but the regex should be
  tightened.

If you find a gap that isn't documented here or in `specs/security.md`,
please report it via the channel below.

## Reporting a vulnerability

**Do not open a public GitHub issue for a security problem.**

Use GitHub's private security advisory form:
<https://github.com/Pawansingh3889/OpsMind/security/advisories/new>

A good report contains:

1. **What you found** — one-sentence description.
2. **Reproduction steps** — the exact question, SQL, or configuration
   that triggers it.
3. **Impact** — what an attacker could do. Speculation is fine; say so.
4. **Suggested fix** — optional. A test case or patch is gold.

## Coordinated disclosure

We follow **90-day coordinated disclosure** by default:

1. You report privately via the advisory form.
2. We confirm receipt and start investigating.
3. We push a fix and a test covering the vulnerability.
4. We publish the advisory (crediting you unless you ask otherwise)
   **90 days after receipt**, or sooner if the fix is deployed and
   you agree.

Longer timelines are available on request when the fix needs upstream
changes (e.g. in Ollama or sqlparse).

## Scope

**In scope:**

- `modules/`, `api.py`, `app.py`, `mcp_servers/`
- `scripts/seed_demo_db.py`
- SQL generated by the agent and validated by `sql_validator`
- Configuration parsing (`schema.yaml`, env vars)
- Docker setup (`Dockerfile`, `docker-compose.yml`)

**Out of scope:**

- Vulnerabilities in upstream dependencies — report those upstream and
  link the advisory here so we can pin the fixed version.
- Issues that require the attacker to already have shell access to the
  host, `sudo`, or the ability to edit `schema.yaml` at runtime. Those
  are containment failures, not OpsMind bugs.
- Social engineering of the maintainer or other contributors.
- Gemma 3 model weight licensing — that's Google's terms, not ours.
